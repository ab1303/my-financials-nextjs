# Development Standards — Low-Level Design

## Naming Conventions

### Files & Folders

| Type | Pattern | Example |
|------|---------|---------|
| Feature folder | kebab-case | `src/features/transaction-management/` |
| Component file | PascalCase | `src/components/TransactionTable.tsx` |
| Service file | kebab-case.service | `src/server/services/transaction.service.ts` |
| Type file | kebab-case.types | `src/types/transaction.types.ts` |
| Schema file | zod schema files | `src/_schema.ts` (co-located with feature) |
| Test file | {name}.test.ts | `src/components/TransactionTable.test.ts` |
| Layout file | layout.tsx | `src/app/cashflow/layout.tsx` |
| Page file | page.tsx | `src/app/cashflow/income/page.tsx` |
| Route handler | route.ts | `src/app/api/transactions/route.ts` |

### Code Conventions

**Functions & Variables**:
- Use camelCase for all function and variable names
- Use descriptive names (avoid `x`, `data`, `temp`)
- Prefix boolean predicates with `is`, `has`, `can`: `isActive`, `hasError`, `canSubmit`
- Prefix event handlers with `handle`: `handleClick`, `handleSubmit`
- Prefix data fetchers with `fetch` or `get`: `fetchTransactions`, `getBalance`

**TypeScript Interfaces & Types**:
- Use PascalCase for interfaces and types
- Prefix types that represent props with the component name: `TransactionTableProps`
- Prefix types that represent domain models with the domain: `TransactionRecord`, `DonationPayment`
- Use `Record<K, V>` over object literals for dictionaries
- Suffix types representing API responses with `Response` or `Result`: `TransactionResponse`

**React Components**:
- Name components with PascalCase matching the filename
- Use `{Component}Props` for prop interfaces
- Use `{Component}State` for complex state types (if not using `useState`)
- Mark Client Components with `'use client'` at the top
- Co-locate private types/schemas in `{feature}/_types.ts` and `{feature}/_schema.ts`

### Database Schema Naming

| Type | Pattern | Example |
|------|---------|---------|
| Model name | PascalCase, singular | `Transaction`, `DonationPayment` |
| Field name | camelCase | `transactionId`, `createdAt`, `amount` |
| Relation name | camelCase, plural for collections | `transactions`, `payments` |
| Unique constraint | On business key or synthetic ID | `@unique` on `transactionId` for FK relations |
| Index name | Auto-generated by Prisma | (Prisma handles this) |

## Module Organization

### Feature-Based Structure

All features are organized as self-contained modules under `src/features/{feature-name}/`:

```
src/features/transaction-management/
  ├── _components/          # Components used only by this feature
  │   ├── TransactionTable.tsx
  │   └── TransactionForm.tsx
  ├── _schema.ts           # Zod schemas for forms/validation
  ├── _types.ts            # TypeScript types specific to this feature
  ├── actions.ts           # Server Actions for mutations
  ├── page.tsx             # Page component (if route)
  ├── layout.tsx           # Layout (if route with children)
  └── README.md            # Feature documentation
```

### Shared Components

Components used by 2+ features go in `src/components/`:

```
src/components/
  ├── ui/                  # shadcn/ui or unstyled primitives
  │   ├── button.tsx
  │   ├── modal.tsx
  │   └── ...
  ├── Table.tsx            # Shared TanStack Table wrapper
  ├── Form.tsx             # Shared form wrapper
  └── ...
```

### Services & Utilities

Backend services are organized by concern:

```
src/server/
  ├── services/            # Business logic services
  │   ├── transaction.service.ts
  │   ├── calendar.service.ts
  │   └── ...
  ├── api/                 # tRPC routers
  │   ├── routers/
  │   │   ├── transaction.router.ts
  │   │   └── ...
  │   └── trpc.ts          # tRPC setup
  └── auth/                # Auth configuration
      └── ...
```

## Best Practices

1. **Keep components small** — If a component exceeds 300 lines, split it
2. **Use Server Components by default** — Fetch data in Server Components, pass to Client Components as props
3. **Use react-hook-form + Zod** — For all forms with validation
4. **Use TypeScript strictly** — No `any` types; use `unknown` with narrowing instead
5. **Test business logic** — Services and hooks should have unit tests
6. **Use meaningful commit messages** — Follow Conventional Commits (feat:, fix:, refactor:, etc.)
7. **Document public APIs** — Add JSDoc comments to exported functions and types
8. **Avoid circular dependencies** — Use barrel exports (index.ts) to break cycles

## Validation Checklist

- [ ] All new files follow naming conventions
- [ ] Feature modules are self-contained under `src/features/{feature}/`
- [ ] Shared components are in `src/components/`
- [ ] Services follow business logic organization
- [ ] No hardcoded strings or magic numbers (extract to constants)
- [ ] Types are co-located or shared appropriately
- [ ] Tests exist for business logic
